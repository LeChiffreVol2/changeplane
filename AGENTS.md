# Prototype Instructions

## Current launch decisions (July 21, 2026)

- Public observe onboarding is self-serve when `CHANGEPLANE_SELF_SERVE_ENABLED=true`. This supersedes the earlier public-root canary restriction for onboarding only: GitHub.com personal accounts and organizations, including Enterprise Cloud organizations, may install the repository-scoped GitHub App, choose any writable repository granted to an eligible installation, and create one protected setup pull request. GitHub Enterprise Server is not supported yet.
- A returning user may have more than one eligible GitHub App installation. Repository discovery must combine only the installations visible to that signed-in user, retain the installation-to-repository binding internally, and never fall back to broad repository access.
- Bring-your-own OpenAI credentials are a per-repository option for personal and organization users, not an enterprise-only feature. The browser sends a key once; ChangePlane verifies it, encrypts it with GitHub's repository public key, stores only `OPENAI_API_KEY` as an Actions Secret, and clears the field. Never store the key in localStorage, logs, API responses, cookies, or a ChangePlane database.
- Self-serve onboarding never broadens authority implicitly. Autonomous repair requires the exact behavioral check, verified repository BYOK, reviewed managed setup PR, dedicated App identity, signed ledger, and fresh exact-head recheck; every missing gate fails closed.
- Product copy should read like a minimal YC-grade developer product: lead with the user outcome and one next action, use plain language, and move exact-head, authority, and runtime detail into progressive disclosure. Preserve the exact hero `Keep GitHub. Let agents ship.` and the existing deep-teal frame/warm off-white visual system.

Run the local server yourself and open the preview in the browser available to this environment. Do not give the user server-start instructions when you can run it.

Before making substantial visual changes, use the Product Design plugin's `get-context` skill when the visual source is unclear or no longer matches the current goal. When the user gives durable prototype-specific design feedback, preferences, or decisions, record them in `AGENTS.md`.

When implementing from a selected generated mock, treat that image as the source of truth for layout, component anatomy, density, spacing, color, typography, visible content, and hierarchy.

## Product direction

- July 21 autonomous-release decision supersedes the earlier public-canary-only onboarding constraint: self-serve GitHub App onboarding is available to individual and organization/enterprise installations. Users connect GitHub, choose one writable repository, bind one exact behavioral check, save their own OpenAI key directly as an encrypted repository Actions Secret, and merge one protected setup PR.
- The self-serve normal path is the managed autonomous harness: GitHub events trigger exact-head evaluation, fixable failed evidence may receive at most two bounded attempts within 15 minutes, and protected, ambiguous, stale, provider-failed, or exhausted cases stop for a human. Scope-only users remain in observe mode.
- Autonomous mode must remain `model proposes; deterministic harness decides; trusted controller applies`. Repository policy is read only from the trusted default branch; the proposal job receives no GitHub credential; App-signed ledger entries and one-time exact-repository installation tokens authorize apply; only a fresh exact-head recheck may publish PASS. GitHub remains the merge authority.
- Do not add a queue, database, proprietary agent workspace, or generalized orchestrator for the autonomous release. Vendor the small Action/controller/repair helpers into the setup PR, keep runtime configuration auditable through `.changeplane.json`, and use GitHub Actions, Checks, comments, artifacts, refs, and repository secrets as the operational substrate.
- `ChangePlane / review` is an independent, BYOK-gated advisory Check: inspect only the exact diff, validate and deduplicate at most five changed-line findings, and never approve, repair, certify, or contribute PASS.
- `.changeplane/assurance.md` is repository-owned assurance memory and the policy-pack surface. Read it only from the trusted default branch; treat it as review context, never behavioral evidence or authority.
- Agent handback is vendor-neutral and GitHub-native. The Action output and exact-head receipt comment may carry bounded findings to Codex, Cursor, Trae, Copilot, OpenSWE, or another agent, but never carry credentials or controller grants.
- Include an existing preview only when its GitHub Deployment SHA matches the evaluated head. ChangePlane does not host previews.
- GitHub Merge Queue evaluates `merge_group` as its own exact revision and publishes `ChangePlane / guard`; queue runs never dispatch model review, proposal, repair, apply, or handback work.

- Selected direction: option 2, a Cursor-inspired hybrid with a quiet deep-teal textured frame and warm off-white product surface.
- Keep the product global, minimal, and operational. Primary users are developers and PMs reviewing agent-authored pull requests.
- Avoid generic AI gradients, dashboard-card grids, neon accents, marketing copy, and unexplained "world model" language in the product UI.
- The VC walkthrough must be fully clickable from sign-in through a GitHub Check decision, use realistic product copy, and never label the experience as a demo in the UI.
- Product category: a GitHub-native agent delivery control plane whose durable wedge is independent assurance. ChangePlane complements GitHub/GitLab/Origin; it does not replace the forge or coding-agent runtime.
- Primary positioning: `Keep GitHub. Let agents ship.` The promise is no forge migration, any repair agent behind one bounded contract, and human involvement only for exceptions.
- `No more localhost:3000` is a marketing punchline about people pasting an unshareable local URL to teammates. It is not a requirement to build preview hosting. Reuse an existing GitHub Deployment/Vercel preview and bind its URL to the exact head SHA in the Change Receipt.
- From the user's perspective agents handle Git; internally the model only proposes a patch. A trusted controller validates exact head, paths, policy, evidence, and attempt budget before any commit or push, and the model never receives merge or Check authority.
- GitHub MCP is optional read-only context plumbing for a repair worker, not a product moat or verifier. Keep GitHub App/API as the control plane and do not add MCP write tools to the pilot.
- Present the agentic backbone as `model proposes; deterministic harness decides; trusted controller applies`. The bundled OpenAI adapter defaults to GPT-5.6 Luna for bounded patch proposals, gives that job no forge write permission, validates the candidate again in a clean apply job, and explicitly dispatches an exact-head recheck. The model transport is replaceable by reviewed code and the revision-bound harness is the product moat.
- The target normal path is autonomous: bind the change contract, run deterministic evidence, return fixable findings to the original agent for at most two attempts / 15 minutes, re-verify, then publish the required Check. GitHub remains responsible for branch protection and merge; humans see protected, ambiguous, exhausted, or blocked exceptions only.
- Production onboarding is GitHub App-first: install on a chosen account/organization, verify the returned installation through PKCE user authorization, list only writable repositories in that installation, and create one atomic managed setup PR. Keep broad GitHub OAuth only as an explicitly labeled observe fallback. Never write the default branch directly or present fixture workspace state as a connected account.
- Publish `ChangePlane / guard` explicitly on the evaluated exact SHA. Present enforcement only after the dedicated GitHub App, expected-App check source, App-signed attempt ledger, live sandbox validation, and negative fail-closed cases pass.
- Pilot architecture stays deliberately small: the GitHub connector, one GitHub Action, one pure evaluator, GitHub Checks/comments as the audit surface, and optional bounded repair adapters. Do not add a database, queue, dashboard service, merge service, or proprietary agent runtime before design-partner demand.
- GPT-5.6 Luna is the default proposal and advisory-review model; Terra and Sol remain allowlisted choices. Keep model access and forge write permission in separate jobs, run helpers only from a trusted-base checkout, treat the pull-request checkout as data, validate the exact head and granted paths before applying a patch, and never let a model issue `PASS`. Compatibility workers stay behind the same adapter contract and out of the launch UI.
- A trusted apply must never push with the workflow `GITHUB_TOKEN`, because that push does not start fresh GitHub Actions. After the App-signed grant is claimed and revalidated, reserve one generation-bound push credential in the signed ledger, then mint one exact-repository installation token with only Contents write immediately before a force-with-lease push. Keep it in runner temp, disable checkout credential persistence, clean it up on every exit, and rely on the resulting `pull_request synchronize` event to start fresh CI and the exact-head guard.
- Keep Grok Build behind the same repair-adapter boundary until its early-beta runtime passes a strict-sandbox canary; its sandbox-off default is not acceptable for the production pilot.
- Funding posture for the pilot: `ChangePlane Managed` is a visible but disabled subscription/control-plane reservation. Do not claim managed model execution, metering, or billing until those server-side systems exist.
- Repository BYOK is the usable funding path for personal and organization users. Plaintext provider keys must never enter localStorage, logs, responses, or a ChangePlane database; send them once through the authenticated same-origin API, encrypt with GitHub's repository public key, store only as an Actions Secret, and clear the browser field after every attempt.
- The MVP has no ChangePlane CLI or TUI. A platform lead installs once through the web; developers continue using their existing IDE or coding-agent CLI, and GitHub is the daily decision and exception surface.
- GPT-5.6 Luna is the current OpenAI canary for repository BYOK and bounded proposal/review adapters. BYOK is verified before it is encrypted as `OPENAI_API_KEY`; the model remains neither verifier nor moat, and ChangePlane Managed stays disabled until isolation, budgets, metering, and billing exist.
- Files under `.github/workflows` are active by definition. Keep observe/repair templates under `examples/` until deliberately installed; production CI is the only repository workflow that should be active by default.
- Production API responses carry a request ID and emit only structured redacted metadata. Never log request bodies, cookies, OAuth tokens, repository names, provider keys, or upstream response bodies.
- Controlled repair canaries must set `CHANGEPLANE_CANARY_REPOSITORY` to the exact disposable `owner/repository`; canary repair routes reject a mismatch before GitHub access. Self-serve repository discovery remains verified-installation scoped. Never use the RouteThai repository as a canary target.
- Operating-cost constraint: use included/free infrastructure and repository-native state. Users fund OpenAI usage through repository BYOK; ChangePlane Managed spend remains disabled. Treat free-tier quotas as hard fail-closed limits.
- Hosting direction is Vercel-only and the current free deployment is a fixed phase constraint. Do not pause product work, compare paid plans, propose alternate hosts, or spend product updates on hosting changes.
- Keep controlled-canary repair authority bound fail-closed to the exact disposable repository. Public onboarding may reach other repositories only through the signed-in user's verified GitHub App installations and may not reuse canary authority.
- The public root offers GitHub App onboarding when self-serve is enabled and always keeps the fictional workspace available without sign-in. Do not present replay state as a connected account.
- Agent-scale strategy is event-driven admission control, not raw Git throughput: use one GitHub Actions concurrency lane per pull request, cancel stale heads, honor GitHub rate-limit headers, and fail closed before expensive evidence or model work.
- Do not add a ChangePlane queue or polling service for speculative scale. Move to GitHub App installation tokens and webhooks first; add durable delivery infrastructure only after measured pilot backlog or repair-delivery requirements justify it.
- GitHub Merge Queue support is guard-only: evaluate `merge_group` as its own exact revision and publish `ChangePlane / guard`. Never dispatch repair or model review for queue events.
- Do not reposition ChangePlane as an AI Agent Orchestrator, proprietary virtual workspace, Git hosting layer, or AST merge engine. Agent runtimes own workspaces; GitHub owns branches, squash merge, merge queue, and history.
- Never claim zero data retention, SOC 2, GDPR compliance, Origin-equivalent speed, 60-second enterprise approval, or support for 100+ agents without audited controls and measured production evidence. Processing repository code on ChangePlane infrastructure would expand, not remove, the compliance boundary.
- Concurrent open-PR file overlap is a bounded advisory only: inspect at most the 20 most recently updated open pull requests and 100 files per pull request in one GraphQL query, show no more than five collisions, and never change PASS or auto-merge based on incomplete overlap data.
- Keep observe onboarding to the three required steps only: connect GitHub, choose one repository, and merge one setup PR. Provider funding is optional after observe and must not block or dominate first-run setup.
- Do not use calendar timelines or elapsed-day claims to promote observe into enforce. Promotion is evidence-gated only. Operational timeouts and the 15-minute/two-attempt repair budget remain safety invariants, not roadmap promises.
- Repository selection must run a read-only safety preflight before enabling the setup PR. Show the exact no-impact boundary: no direct default-branch write, no merge/deploy blocking, no repair dispatch, no PR-head execution, no provider-secret access, and no overwrite of reserved paths.
- Ordinary pull requests must not require a developer to hand-author a ChangePlane marker. When no explicit contract exists, bind the first observed head's exact paths and title automatically, persist that contract in the trusted receipt comment, and treat later path expansion as scope drift. Route changes above the 50-path automatic-contract limit to a clear split-or-declare exception.
- The primary walkthrough use case is a checkout retry change where deterministic race evidence reproduces a duplicate charge, returns the exact failure to the original coding agent, accepts one bounded in-contract patch, re-verifies a new head, and publishes the GitHub Check with zero human actions. Use this concrete evidence-repair story instead of trivial scope drift.
- A public share link should open with one working `Open example workspace` action. Do not lead with a disabled GitHub control; clearly distinguish the fictional workspace and controlled assurance canary from connected GitHub state.
- When the GitHub connector is not configured on a public deployment, fall back to that clearly fictional example workspace on the root URL instead of rendering a disabled primary action. Once configured, the same root switches to the real one-time GitHub installer.
- The connector is GitHub App-first when `GITHUB_APP_SLUG` is configured and restricts the picker to the verified installation. The shared broad OAuth identity at `changeplane.vercel.app` remains an observe-only fallback until the production GitHub App registration is configured; never create an OAuth app per customer or represent the fallback as enforce-ready.
- The normal user journey has no per-PR handoff to the ChangePlane web app. A platform lead installs once; every later `pull_request` update triggers the assurance loop automatically, while developers remain in their coding agent and GitHub. Interactive controls in the fictional workspace are canary replay controls only and must say so.
- Treat GitHub beginners and vibe coders as a first-class self-serve audience without turning ChangePlane into an IDE: no CLI is required, onboarding remains connect → choose repository → merge one setup PR, and every blocked state must name the plain-language reason, the safe consequence, and one next action. Keep Git-specific detail available as secondary evidence for platform engineers.
- Lead the product differentiation with independent revision-bound assurance, not a claim that GitHub or Cursor lack repair. Deterministic evidence may return a fixable finding to any coding agent under a bounded contract, but only a separate trusted controller and exact-head re-verification may publish PASS. GitHub stays the forge and merge authority.
- The public production connector is the GitHub App `changeplane-guard`; keep its installation repository-scoped and require a protected pull request for every production deployment. The source repository is public and source-visible for commercial product transparency, but remains proprietary and `UNLICENSED`; public visibility does not relax secret scanning, branch protection, release provenance, or copyright controls.
- Qualify self-serve users before connection: ChangePlane is most immediately valuable for GitHub projects that already run meaningful automated tests on pull requests. The setup PR must teach how to bind an exact check name and publisher, and the UI must clearly distinguish a scope-only receipt from behavioral evidence.
- The installer asks explicitly whether the first receipt should bind one existing behavioral check or remain scope-only. Behavioral evidence is recommended and written atomically into the setup PR; scope-only must be a deliberate choice that says it does not prove the code works.
